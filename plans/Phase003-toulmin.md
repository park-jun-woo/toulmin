# Phase 003: Graph Builder API — 함수 기반 그래프 조립 + Rule 재활용 (구현 완료)

## 목표

함수 자체를 식별자로 사용하는 Graph Builder API 구현.
Rule 함수와 defeats 관계를 분리하여 동일 함수를 여러 그래프에서 재활용 가능하게 한다.
`toulmin graph` 커맨드를 YAML → Go Graph Builder 코드 생성으로 전환한다.

## 배경

Phase 001-002에서 defeats 관계가 `RuleMeta.Defeats`에 박혀있어,
동일 함수를 다른 그래프에서 다른 defeats 관계로 사용할 수 없다.
함수와 그래프 구조를 분리하면 Rule 재활용이 가능해진다.

### 핵심 설계 결정

1. **함수가 곧 식별자**: `reflect`/`runtime.FuncForPC`로 함수 포인터에서 이름 추출. 문자열 이름 불필요
2. **Qualifier는 Rule당**: 툴민 원래 모델의 Qualifier(Claim 확신도)를 Rule 초기 가중치로 재배치 (논문 §3.3). 기본값 1.0 (원래 모델 존중)
3. **Backing은 함수 어노테이션**: `//rule:backing "..."` — 그래프가 아닌 함수에 부착
4. **Defeat는 그래프 엣지**: Qualifier 없음, 관계 선언일 뿐
5. **YAML → Go 코드 생성**: 그래프 구조를 YAML로 정의하고, `toulmin graph` 커맨드로 Graph Builder 코드를 생성

## 범위

### 포함

1. **Graph Builder**: 체이닝 API — `NewGraph`, `Warrant`, `Rebuttal`, `Defeater`, `Defeat`
2. **함수 식별자 추출**: `reflect.ValueOf(fn).Pointer()` → `runtime.FuncForPC` → 함수명
3. **Evaluate 통합**: `graph.Evaluate(claim, ground)` — claim/ground는 평가 시점에 주입
4. **YAML → Go 코드 생성**: `toulmin graph <yaml>` — YAML 그래프 정의를 Go Graph Builder 코드로 변환
5. **기존 엔진과 공존**: Phase 001의 `Engine.Register` + `Engine.Evaluate`는 유지

### 제외

- 기존 `//rule:` 어노테이션의 `defeats=` 제거 — Phase 002 호환 유지

## API 설계

### 사용 예시

```go
votingGraph := toulmin.NewGraph("voting").
    Warrant(IsAdult, 1.0).
    Warrant(IsCitizen, 1.0).
    Rebuttal(HasCriminalRecord, 1.0).
    Defeat(IsAdult, HasCriminalRecord)

verdict := votingGraph.Evaluate(claim, ground)
```

### Rule 재활용

```go
// 같은 IsAdult 함수, 다른 그래프, 다른 defeats 관계
votingGraph := toulmin.NewGraph("voting").
    Warrant(IsAdult, 1.0).
    Rebuttal(HasCriminalRecord, 1.0).
    Defeat(IsAdult, HasCriminalRecord)

contractGraph := toulmin.NewGraph("contract").
    Warrant(IsAdult, 1.0).
    Rebuttal(IsBankrupt, 1.0).
    Defeat(IsAdult, IsBankrupt)
```

### Qualifier 기본값

```go
// 툴민 원래 모델 존중 — Qualifier 미지정 시 1.0
votingGraph := toulmin.NewGraph("voting").
    Warrant(IsAdult).        // qualifier = 1.0
    Warrant(IsCitizen, 0.7). // qualifier = 0.7 (확장 사용)
```

## YAML → Go 코드 생성

### 입력: YAML 그래프 정의

```yaml
graph: voting
rules:
  - name: IsAdult
    role: warrant
    qualifier: 1.0
  - name: IsCitizen
    role: warrant
    qualifier: 0.7
  - name: HasCriminalRecord
    role: rebuttal
    qualifier: 1.0
defeats:
  - from: IsAdult
    to: HasCriminalRecord
```

### 출력: 생성된 Go 코드

```go
// Code generated by toulmin. DO NOT EDIT.
package <소스 패키지명>

import "github.com/park-jun-woo/toulmin/pkg/toulmin"

var VotingGraph = toulmin.NewGraph("voting").
    Warrant(IsAdult, 1.0).
    Warrant(IsCitizen, 0.7).
    Rebuttal(HasCriminalRecord, 1.0).
    Defeat(IsAdult, HasCriminalRecord)
```

### CLI

- `toulmin graph <yaml>` — YAML 읽기 → Go Graph Builder 코드 생성
- `--output` 플래그로 출력 경로 지정 (기본: `<yaml 디렉토리>/graph_gen.go`)
- `--dry-run` 플래그로 stdout 출력

## 단계

### Step 1: 함수 이름 추출

- `reflect.ValueOf(fn).Pointer()` → `runtime.FuncForPC(ptr).Name()`
- 패키지 경로 제거, 함수명만 반환
- 클로저/메서드 등 예외 케이스 처리

### Step 2: GraphBuilder 구조체

- `NewGraph(name string) *GraphBuilder`
- `Warrant(fn func(any, any) bool, qualifier ...float64) *GraphBuilder`
- `Rebuttal(fn func(any, any) bool, qualifier ...float64) *GraphBuilder`
- `Defeater(fn func(any, any) bool, qualifier ...float64) *GraphBuilder`
- `Defeat(from func(any, any) bool, to func(any, any) bool) *GraphBuilder`
- 내부에서 함수 포인터로 이름 추출, RuleMeta 생성 + defeats 엣지 별도 저장

### Step 3: Evaluate 통합

- `GraphBuilder.Evaluate(claim any, ground any) []EvalResult`
- builder 내부의 rule 목록 + defeats 엣지로 RuleGraph 구성
- 기존 `BuildSubgraph` + `CalcAcceptability` 재사용
- claim/ground는 평가 시점에 주입

### Step 4: YAML 파서

- YAML 그래프 정의 파싱 → 내부 GraphDef 구조체
- graph 이름, rules (name, role, qualifier), defeats (from, to) 추출
- qualifier 미지정 시 기본값 1.0

### Step 5: Go 코드 생성

- GraphDef → Go Graph Builder 코드 문자열 생성
- `// Code generated by toulmin. DO NOT EDIT.` 헤더
- package명은 YAML 파일 위치의 Go 패키지를 따름
- 출력 파일: `graph_gen.go`

### Step 6: CLI 전환

- `toulmin graph <yaml>` — YAML 파싱 → 검증 → Go 코드 생성
- 기존 `toulmin graph <dir>` (소스 스캔 방식)은 deprecate

### Step 7: 문서 갱신

- `manual-for-ai.md` — Graph Builder API, YAML 그래프 정의, 변경된 `toulmin graph` 커맨드 반영
- `README.md` — 사용 예시를 Graph Builder API 기반으로 갱신, YAML 그래프 정의 설명 추가

### Step 8: 테스트

- 동일 함수 다른 그래프 재활용 검증
- Qualifier 기본값 1.0 검증
- Defeat 엣지가 그래프 레벨에서만 적용되는지 검증
- 기존 Engine API와 결과 동일성 검증
- YAML → Go 코드 생성 결과가 컴파일되는지 검증
- 생성된 코드로 Evaluate 결과가 수동 작성과 동일한지 검증

## 검증 기준

1. 같은 함수를 두 그래프에서 다른 defeats 관계로 사용 가능
2. Qualifier 미지정 시 기본값 1.0
3. Defeat는 그래프 엣지일 뿐, 함수에 영향 없음
4. `GraphBuilder.Evaluate` 결과가 기존 `Engine.Evaluate`와 동일 조건에서 동일
5. 함수명 추출이 일반 함수/패키지 함수에서 정확히 동작
6. YAML 그래프 정의에서 생성된 Go 코드가 컴파일된다
7. defeats 참조가 존재하지 않는 rule을 가리키면 에러
